## 詳細リスク分析

### 1. ユーザー獲得リスク 🔴

**リスク内容:**
ギフト管理アプリはニーズが顕在化しにくく、「なくても困らない」カテゴリに分類されがち。アプリストアでの自然流入だけでは成長が限定的。

**対策:**
- **バイラルループの設計** — ウィッシュリストを共有された人が自然にアプリをDLする仕組み
- **季節イベント施策** — クリスマス・バレンタイン・母の日など、ギフト需要のピーク時に集中マーケ
- **LINEミニアプリ版** — アプリDL不要でウィッシュリスト閲覧可能にし、ファネルを広げる
- **KPI**: 招待経由DL率 30%以上を目標

### 2. EC連携の技術的課題 🔴

**リスク内容:**
各ECサイトのメタデータ取得は、サイト構造変更で頻繁に壊れる。アフィリエイトリンクの管理も複雑。

**対策:**
- **OGPベースの取得を基本** — 最も安定的な手法
- **フォールバック** — OGP取得失敗時はユーザー手動入力に切り替え
- **段階的EC対応** — Amazon → 楽天 → Yahoo! の順で対応、一気にやらない
- **メタデータ取得の監視** — 日次で取得成功率をモニタリング、閾値割れでアラート

### 3. 個人情報保護リスク 🟡

**リスク内容:**
ギフト履歴は高度にプライベートな情報。漏洩時のブランドダメージは致命的。

**対策:**
- **Supabase RLS（Row Level Security）** — データベースレベルでのアクセス制御
- **E2E暗号化** — メッセージカードの内容は端末間で暗号化
- **個人情報の最小化** — 不要な情報は取得しない
- **プライバシーポリシーの透明性** — ユーザーが自分のデータを完全削除できる機能
- **定期セキュリティ監査** — 四半期ごとに外部監査を実施

### 4. 競合参入リスク 🟡

**リスク内容:**
Amazon、LINEなどの大手プラットフォーマーが類似機能を実装する可能性。LINE GIFTは既に存在。

**対策:**
- **データのポータビリティ×ロックイン** — ユーザーのギフト履歴データは他では得られない資産
- **UXでの差別化** — 大手の「おまけ機能」ではなく、ギフト特化のディープなUX
- **コミュニティ形成** — ギフト好きのコミュニティ、レビュー、ランキング
- **ニッチ→拡大** — まず「ウィッシュリスト共有」で圧倒的No.1を取り、そこから拡張

### 5. 課金転換率リスク 🟡

**リスク内容:**
無料で十分満足するユーザーが多く、Premium/Familyへの転換が進まない。

**対策:**
- **無料版の制限を適切に設計** — 使いすぎると自然にPremiumが必要になる設計
- **トライアル期間** — 新規ユーザーに14日間のPremium体験を提供
- **価格テスト** — ¥380〜¥680の範囲でA/Bテスト
- **法人プランの早期展開** — B2Cの課金に依存しすぎない収益構造

## 法的リスク

### 景品表示法
- アフィリエイトリンクの明示義務 → アプリ内で「提携リンク」表示
- 価格表示の正確性 → ECサイトからのリアルタイム取得

### 個人情報保護法
- 第三者提供の制限 → ユーザー同意取得フロー
- 越境データ移転 → サーバーは日本リージョンに限定（初期）

### 資金決済法
- グループギフトの集金機能 → 資金移動業の登録要否を弁護士に確認
- エスクロー型にすることで規制回避の可能性

## モニタリング体制

| 指標 | 閾値 | アクション |
|------|------|-----------|
| メタデータ取得成功率 | < 90% | EC連携チームにアラート |
| サーバー応答時間 | > 500ms | インフラチームにエスカレーション |
| DAU/MAU比 | < 20% | プロダクトチームで改善施策検討 |
| 課金チャーン率 | > 10%/月 | 解約理由分析、機能改善 |
| カスタマーサポート対応時間 | > 24h | 人員増強検討 |
