# リスク分析

## 事業リスク

### 1. データ取得リスク 🔴 高
| 項目 | 詳細 |
|------|------|
| **リスク** | 中古車データのAPI提供が限定的。カーセンサー/グーネットの公式APIは法人契約が必要 |
| **影響** | サービスの根幹であるデータが取得できない |
| **対策** | ① 公式API法人契約を早期に交渉 ② 初期はスクレイピング＋手動データ投入で検証 ③ ユーザー投稿型データ（実購入価格の共有）で独自DB構築 |

### 2. 価格精度リスク 🟡 中
| 項目 | 詳細 |
|------|------|
| **リスク** | 中古車は1台ごとに状態が異なり、価格の「相場」が不正確になりやすい |
| **影響** | ユーザーの信頼低下、ミスリードによるクレーム |
| **対策** | ① 価格帯（レンジ）で表示し、特定価格を保証しない ② 「参考価格」である旨を明記 ③ 実在する在庫データへのリンクで実態を確認可能に |

### 3. 競合参入リスク 🟡 中
| 項目 | 詳細 |
|------|------|
| **リスク** | カーセンサーやグーネットが同様の比較機能を自社実装する可能性 |
| **影響** | データソースを持つプラットフォーマーに勝てない |
| **対策** | ① 先行者優位でブランド確立 ② UX/体験の差別化（車格スコアという独自指標） ③ コミュニティ機能（実オーナーのレビュー）で参入障壁構築 |

### 4. 法的リスク 🟢 低
| 項目 | 詳細 |
|------|------|
| **リスク** | 景品表示法、特定商取引法への抵触（価格表示の正確性） |
| **影響** | 行政指導、サービス停止 |
| **対策** | ① 「比較情報提供サービス」として位置づけ、売買の仲介はしない ② 免責事項の明記 ③ リリース前に弁護士レビュー |

## 技術リスク

### 5. スケーラビリティ 🟢 低
- Vercel + Supabase構成なら初期はスケール問題なし
- 車両データのキャッシュ戦略が重要（毎回API呼び出しは非効率）

### 6. AI精度（車格スコア） 🟡 中
- 車格の定義が主観的。「BMW 3シリーズとクラウンはどちらが上か」は人による
- 対策: 客観指標（車両価格、サイズ、排気量、装備数）をベースにしつつ、ユーザー投票で補正

## リスクマトリクス

```
影響大 │  [1.データ取得]
       │           [3.競合参入]
       │  [2.価格精度]    [6.AI精度]
影響小 │  [4.法的]  [5.スケール]
       └──────────────────────────
        発生率低          発生率高
```

## 最優先アクション
1. **カーセンサーAPI法人契約の調査・申請**（1週間以内）
2. **MVP段階では手動データで検証**し、データ取得問題を後回しにする
3. **車格スコアのロジックをユーザーテストで検証**
